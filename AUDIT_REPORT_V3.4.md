## 4. 进一步改进建议 (Refusal to Flatter)

### ⚠️ 痛点 1：Issue 去重逻辑过于简陋
- **现象**：11 个缺陷被去重后仅剩 1 个 Issue。
- **理由**：目前的去重仅基于 `root_cause_analysis` 的前 50 个字符。由于 Agent 4/5 生成的分析报告开头往往包含类似的 `[Confidence: 0.80] [CoT:...` 模板，导致大量本质不同的 Bug 被误杀。
- **建议**：引入 **Semantic Deduplication**。使用向量相似度对比整个 `root_cause_analysis` 段落，或基于 `case_id` 的业务分类进行智能去重。

### ⚠️ 痛点 2：MRE 运行环境的隔离度不足
- **现象**：MRE 运行在主进程的 Python 环境中。
- **理由**：如果 MRE 逻辑涉及修改环境变量或安装包，可能会污染测试台。
- **建议**：将 MRE 验证放入 **Docker Sidecar** 容器中运行，实现绝对的环境物理隔离。

### ⚠️ 痛点 3：Token 成本与 CoT 冗余
- **现象**：Agent 1 提取契约时消耗了大量 Token（单次近 7 万）。
- **理由**：Agent 0 爬取的文档全量喂给了 Agent 1，缺乏预处理。
- **建议**：在 Agent 1 之前增加一个 **Content Pruning (内容剪枝)** 步骤，仅保留包含 API 参数、限制、配置相关的 Markdown 段落。
