# AI-DB-QC 技术报告

> **版本**: v4.5 | **日期**: 2026-04-10 | **作者**: AI-DB-QC Team

---

## 摘要

AI-DB-QC (AI-driven Database Quality Control) 是一套基于大语言模型 (LLM) 多智能体 (Multi-Agent) 架构的向量数据库自动化质量检测系统。本报告详细阐述了系统的核心技术方法：**四型缺陷分类法**、**双层有效性模型**、**Multi-Agent 测试流水线**，以及 v4.4 版本的验证结果。

---

## 一、研究背景与动机

### 1.1 问题陈述

AI数据库（尤其是向量数据库）作为衔接AI模型与业务数据的核心枢纽，在生成式AI、多模态应用爆发的背景下成为关键基础设施。然而，AI数据库因多维度技术融合而具备复杂性，易出现：

- **向量计算偏差**: 高维向量运算精度问题
- **语义匹配失真**: 语义检索结果与预期不符
- **契约约束绕过**: 非法请求被错误接受

传统数据库测试方法侧重于结构化数据的语法校验与性能测试，无法覆盖语义校验、向量精度验证等核心需求。

### 1.2 研究目标

1. 设计基于大模型的AI数据库质量测试用例生成方法
2. 构建AI数据库测试用例正确性判断技术
3. 形成适配AI数据库的质量保障工具

## 二、核心技术方法

### 2.1 四型缺陷分类法 (Four-Type Defect Classification)

基于对向量数据库缺陷特征的深入分析，我们提出了**五类缺陷分类体系**：

#### 2.1.1 分类决策树

```
                    L1: 契约合法?
                         |
           ┌─────────────┴─────────────┐
          NO (有Warning)               YES (无Warning)
           |                            |
         执行成功?                    执行成功?
       ┌──┴──┐                      ┌───┴───┐
      YES   NO                     NO     YES
       |     |                      |       |
    Type-1 Type-2              L2通过?  L2通过?
                                      |       |
                                    NO       YES
                                     |        |
                                  Type-2.PF  Oracle通过?
                                             |
                                           NO     YES
                                            |       |
                                         Type-4  无缺陷(Type-3保留)
```

#### 2.1.2 缺陷类型定义

| 类型 | 名称 | 判定条件 | 证据级别 | 典型场景 |
|------|------|----------|----------|----------|
| **Type-1** | 非法成功 (Illegal Success) | `L1=FAIL \| Exec=SUCCESS` | L1 | 非法维度请求绕过校验并被数据库接受 |
| **Type-2** | 诊断不足 (Poor Diagnostics) | `L1=FAIL\|PASS \| Exec=FAIL \| L2=PASS` | L2 | 请求失败但错误信息模糊不充分 |
| **Type-2.PF** | 前置条件失败 (Precondition Failure) | `L1=PASS \| Exec=FAIL \| L2=FAIL` | L2 | Collection 不存在或无数据导致失败 |
| **Type-3** | 传统预言机违规 (Traditional Oracle) | `L1=PASS \| Exec=SUCCESS \| 传统Oracle=FAIL` | L2/L3 | 单调性/一致性异常 |
| **Type-4** | 语义违规 (Semantic Violation) | `L1=PASS \| Exec=SUCCESS \| 语义Oracle=FAIL` | L3 | 搜索结果排序不合理、相关性不足 |

#### 2.1.3 分类意义

- **Type-1**: 揭示数据库契约校验漏洞，属于安全类缺陷
- **Type-2**: 揭示错误诊断能力不足，影响开发者调试效率
- **Type-2.PF**: 区分真正的缺陷与环境/前置条件问题
- **Type-3**: 传统数据库测试可覆盖的缺陷类型
- **Type-4**: AI数据库特有的语义层缺陷，需LLM预言机检测

### 2.2 双层有效性模型 (Dual-Layer Validity Model)

#### 2.2.1 L1 抽象合法性门控 (Abstract Legality Check)

**目的**: 在测试执行前检查请求是否符合数据库的静态契约约束。

**检查内容**:
- `dimension` 是否在 `allowed_dimensions` 范围内
- `metric_type` 是否在 `supported_metrics` 列表中
- `top_k` 是否超过 `max_top_k` 限制

**行为模式**: **默认硬阻断（可配置）** — 对非法 `dimension`/`top_k` 直接拒绝执行；可通过 `agent3.l1_hard_block_illegal_params=false` 切换为 Warning 放行模式（用于专项绕过探测）。

#### 2.2.2 L2 运行时就绪性门控 (Runtime Readiness Check)

**目的**: 检查测试执行所需的运行时资源是否就绪。

**检查内容**:
- Collection 是否存在 (`state.current_collection`)
- 数据是否已注入 (`state.data_inserted`)
- Index 是否已加载

**行为模式**: **硬阻断** — 未就绪直接标记为 Type-2.PF，避免误报。

#### 2.2.3 双层门控的关键作用

| 门控层 | 检查时机 | 检查类型 | 失败后果 |
|--------|----------|----------|----------|
| L1 | 执行前 | 静态契约 | 默认硬阻断（可切换为 Warning） |
| L2 | 执行前 | 动态资源 | 硬阻断 |

双层门控机制确保：
1. **Type-1 检测**: 默认通过“拒绝+结构化违规详情”定位契约越界；如需绕过探测可临时切换 Warning 模式
2. **Type-2 vs Type-2.PF 区分**: L2 门控结果作为关键判定节点

### 2.3 Multi-Agent 测试流水线

#### 2.3.1 架构设计

系统采用 **LangGraph StateGraph** 编排 9 个核心 Agent 的协作流水线：

```
Agent0 (Recon) ──→ Agent1 (Contract) ──→ Agent2 (Generator)
                                              │
                                              ▼
Agent6 (Verifier) ←── Agent5 (Diagnoser) ←── Agent4 (Oracle)
                                              │
                                              ▲
                                        Agent3 (Executor)
                                              │
                                              ▼
                                        Reranker (Cross-Encoder)
```

#### 2.3.2 Agent 职责

| Agent | 职责 | LLM 能力 |
|-------|------|----------|
| **Agent0** | 环境与情报获取 | 文档爬取、Docker 部署 |
| **Agent1** | 场景与契约分析 | 信息抽取、规则形式化 |
| **Agent2** | 混合测试生成 | 语义发散、对抗样本 |
| **Agent3** | 执行与双层门控 | 状态机维护、适配器调用 |
| **Reranker** | 结果重排序 | Cross-Encoder 语义打分 |
| **Agent4** | 预言机协调 | 语义相关性打分、意图比对 |
| **Agent5** | 缺陷诊断与分类 | 决策树遍历、证据综合 |
| **Agent6** | 验证与去重 | MRE 提取、相似度计算 |

#### 2.3.3 多轮自适应循环

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Agent2 ──→ Agent3 ──→ Reranker ──→ Agent4 ──→ Agent5  │
│    ▲                                           │        │
│    │                                           ▼        │
│    └────────────── Feedback ───────────── Agent5        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

每轮测试结束后，Agent5 将薄弱点特征反馈给 Agent2，驱动下一轮用例的定向进化。

### 2.4 契约回退系统 (Contract Fallback System)

#### 2.4.1 问题背景

Agent1 通过 LLM 从文档中提取契约信息时，可能因文档不完整或 LLM 抽取失败导致部分字段为空。

#### 2.4.2 解决方案

`MilvusContractDefaults` 定义了 Milvus 数据库的完整契约默认值：

| 字段类型 | 默认值 | 说明 |
|----------|--------|------|
| `allowed_dimensions` | 30 种维度 | 含极端值用于压力注入 |
| `supported_metrics` | 6 种度量 | L2, IP, COSINE, HAMMING, JACCARD, BM25 |
| `max_top_k` | 16384 | Top-K 上限 |
| `supported_index_types` | 12 种索引 | FLAT, IVF_FLAT, HNSW 等 |

#### 2.4.3 设计原则

- **非侵入式**: 仅对空字段生效，LLM 已提取的有效值不会被覆盖
- **可扩展**: 通过 `FALLBACK_REGISTRY` 支持新数据库
- **可审计**: 回退操作通过 logger 记录

#### 3.2.2 典型缺陷示例

**Type-4 语义违规案例**:
- **Case ID**: L1_NEG_DIM_001
- **场景**: 负数维度请求
- **预期**: 应返回明确的参数错误
- **实际**: 返回模糊的错误信息
- **判定**: Type-2 (诊断不足)

**Type-4 语义违规案例**:
- **Case ID**: L1_POS_HYBRID_RAG_001
- **场景**: 混合检索结果排序
- **预期**: 相关性高的结果应排在前面
- **实际**: 排序不符合语义预期
- **判定**: Type-4 (语义违规)

| 版本 | Type-2.PF 占比 | 问题根因 |
|------|----------------|----------|
| v4.3 | 100% (伪阳性) | L2 门控状态未传播 |

## 四、技术贡献

### 4.1 理论贡献

1. **四型缺陷分类法**: 首次提出面向向量数据库的五类缺陷分类体系
2. **双层有效性模型**: L1 静态契约 + L2 动态资源的双层门控机制
3. **语义预言机**: 基于 LLM 的语义相关性验证方法

### 4.2 工程贡献

1. **Multi-Agent 测试流水线**: 9-Agent 协同的自动化测试框架
2. **契约回退系统**: 确保测试生成永不阻塞的容错机制

## 五、局限性与未来工作

### 5.1 当前局限

1. **跨数据库支持**: 目前仅支持 Milvus
2. **Type-1/Type-3 检测率**: 需进一步优化
3. **性能**: 单次运行耗时 ~15 分钟

### 5.2 未来方向

| 方向 | 优先级 | 预计周期 |

## 六、结论

AI-DB-QC v4.4 成功实现了基于 LLM Multi-Agent 的向量数据库自动化质量检测系统，核心贡献包括：

1. **四型缺陷分类法**提供了系统化的缺陷分类框架
2. **双层有效性模型**有效区分了不同类型的执行失败
3. **Multi-Agent 流水线**实现了端到端的自动化测试

v4.4 验证结果表明，系统能够有效发现 Type-4 语义违规和 Type-2 诊断不足类缺陷，为向量数据库的质量保障提供了新的技术路径。
