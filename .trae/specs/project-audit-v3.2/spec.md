# 项目全量审计与实战评估规范 (v3.2)

## 为什么
项目已完成 v3.2 版本的核心功能开发与可靠性加固。为了验证系统是否真实符合理论框架（契约驱动、双层有效性、四型分类法）并满足工业级测试要求，需要进行一次全流程的实战运行与客观评估。

## 变更内容
- **实战运行**：执行 `main.py` 完整生命周期（目标：Milvus v2.6.12）。
- **流程观察**：重点观察环境侦察、契约提取、用例生成、物理执行、Oracle 判定、缺陷诊断、Issue 生成及自愈流程。
- **文件审计**：验证 `raw_docs.json`、`telemetry.jsonl`、生成的 `.md` 缺陷报告的内容质量。
- **输出评估报告**：基于实战数据，客观分析系统优劣势。

## 影响
- 受影响 specs: 无（本任务为审计性质）。
- 受影响代码: 无（只读运行与观察）。

## 新增需求
### 需求：客观评估报告
报告 SHALL 包含以下章节：
1. **实战运行快照**：包含轮次、Token 消耗、发现缺陷数。
2. **理论框架符合性分析**：评估实际行为是否遵循 L1/L2/L3 契约系统。
3. **Issue 质量专项审计**：
    - **链接有效性**：验证报告中引用的所有 URL 是否真实可访问。
    - **内容相关性**：评估爬取的网页内容与识别到的 Bug 之间是否存在真实的逻辑关联。
    - **复现验证 (MRE Verification)**：对 Issue 中的 Python 代码进行可运行性测试，核实其是否能复现报告所述行为。
    - **证据链完整性**：审查“文档证据 -> 实际行为 -> 根因分析”的逻辑链条是否闭环且无矛盾。
    - **正确性验证**：核对 Issue 中的技术参数、版本号及契约类型是否准确无误。
4. **关键组件审查**：
    - **Agent 0/1**：爬取深度与契约准确度。
    - **Agent 3**：物理执行稳定性与池化效果。
    - **Agent 4/5**：Oracle 推理逻辑与分类准确度。
    - **Agent 6**：Issue 生成的专业度。
5. **技术债务与缺陷发现**：记录运行中的异常、性能瓶颈及逻辑漏洞。
6. **改进建议与理由**：基于实战痛点的优化方向。

#### 场景：审计成功
- **WHEN** 完成一次 10 轮以上的 Fuzzing 实战。
- **THEN** 产出一份结构严谨、数据详实的评估报告。
