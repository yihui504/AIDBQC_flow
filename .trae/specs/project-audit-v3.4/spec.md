# 项目全量审计与实战评估规范 (v3.4)

## Why
在 v3.4 版本中，我们实施了 MRE 验证网关的逻辑重构、自适应爬虫回退机制以及测试语料库的清洗。为了验证这些重构是否彻底解决了前期发现的“虚假复现”和“证据链断裂”问题，并评估语料清洗对降低 Oracle 误判率的效果，我们需要进行新一轮的实战运行与全量审计。

## What Changes
- **实战运行**：执行 `main.py` 完整生命周期（目标：Milvus v2.6.12，至少 3 轮迭代）。
- **专项审计**：
    - **爬虫机制验证**：核查在官方文档受限时，是否成功回退并抓取到有效内容（`raw_docs.json` 不为空）。
    - **语料库纯净度验证**：抽样检查注入的 Mock 数据，确认无干扰性背景文本。
    - **MRE 验证网关核查**：审查生成的 Issue，验证包含代码错误的 MRE 是否被正确标记为 `[INVALID_CODE]` 或 `[FAILED]`，且不再误报为 `[SUCCESS]`。
    - **Oracle 判定准确率**：评估消除噪声数据后，Type-4（语义违规）的判定是否更加精准。
- **输出评估报告**：客观总结 v3.4 版本的实际表现，指出仍存在的瓶颈并提供下一步优化建议。

## Impact
- 影响规范：无。
- 影响代码：无（仅读取运行结果）。

## ADDED Requirements
### Requirement: 客观评估报告 (Audit Report)
报告必须包含：
1. **实战指标**：轮次、Token 消耗、各状态 Issue 数量（SUCCESS, INVALID_CODE, FAILED）。
2. **爬虫回退验证**：记录回退机制的触发情况及文档抓取质量。
3. **MRE 质量与验证网关审查**：分析验证机制拦截虚假复现的有效性。
4. **语料质量与 Oracle 表现**：评估去噪后语义检索的准确性及假阳性率变化。
5. **待改进点与建议**：基于本轮实战的深入剖析与后续演进方向。

#### Scenario: 审计完成
- **WHEN** 完成 3 轮以上实战并审查所有核心产出物。
- **THEN** 在根目录更新或产出 `AUDIT_REPORT_V3.4.md`。
