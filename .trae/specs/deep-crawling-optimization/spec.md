# 深度爬取能力优化规范

## 为什么
当前 AI-DB-QC 的文档爬取能力存在以下局限：
- 仅爬取 DuckDuckGo 搜索结果中的 URL（约 5-10 个页面），没有进行深度递归爬取
- 导致大量"证据缺失"问题，影响缺陷报告的质量
- 文档覆盖不完整，特别是 API 参考、最佳实践、限制说明等深层内容

本优化计划旨在通过配置 Crawl4AI 进行深度为 3 层的递归爬取，大幅提升文档覆盖率，减少"证据缺失"问题。

## 变更内容
- **深度递归爬取**：配置 Crawl4AI 进行深度为 3 层的递归爬取，从官方文档首页开始，逐层提取和爬取链接
- **智能链接过滤**：实现链接过滤机制，只爬取同一域名下的相关文档页面，避免爬取无关内容
- **爬取深度控制**：实现爬取深度控制，确保爬取范围可控，避免过度爬取
- **进度监控**：添加爬取进度监控，实时显示爬取的 URL 数量和深度
- **组件适配**：适配其他组件（如 knowledge_base.py、agent6_verifier.py）以处理更大量的文档数据

## 影响
- 受影响规范：文档爬取流程、证据链验证逻辑、知识库检索逻辑
- 受影响代码：`agent0_env_recon.py`, `knowledge_base.py`, `agent6_verifier.py`, `reference_validator.py`

## 新增需求

### 需求：深度递归爬取
系统 SHALL 支持深度为 3 层的递归爬取，从官方文档首页开始，逐层提取和爬取链接。

#### 场景：成功案例
- **WHEN** Agent 0 爬取 Milvus 官方文档
- **THEN** 系统从 https://milvus.io/docs/ 开始，递归爬取深度为 3 层的所有相关文档页面

#### 场景：深度控制
- **WHEN** 爬取深度达到 3 层
- **THEN** 系统停止继续爬取，返回已爬取的所有文档内容

### 需求：智能链接过滤
系统 SHALL 实现智能链接过滤机制，只爬取同一域名下的相关文档页面。

#### 场景：同域名链接
- **WHEN** 爬取过程中发现链接
- **THEN** 系统只爬取与起始 URL 同一域名下的链接

#### 场景：排除无关链接
- **WHEN** 爬取过程中发现链接
- **THEN** 系统排除以下类型的链接：
  - 静态资源（.css, .js, .png, .jpg 等）
  - 外部链接（不同域名）
  - 锚点链接（#开头）
  - 登录/注册页面

### 需求：爬取进度监控
系统 SHALL 提供爬取进度监控，实时显示爬取状态。

#### 场景：进度显示
- **WHEN** 爬取过程中
- **THEN** 系统实时显示：
  - 当前爬取深度
  - 已爬取 URL 数量
  - 待爬取 URL 数量
  - 当前正在爬取的 URL

### 需求：文档去重
系统 SHALL 实现文档去重机制，避免重复爬取相同 URL。

#### 场景：URL 去重
- **WHEN** 爬取过程中发现已爬取过的 URL
- **THEN** 系统跳过该 URL，不进行重复爬取

## 修改需求

### 需求：增强型文档存储
`agent0_env_recon.py` SHALL 将递归爬取的所有文档内容存储到 `raw_docs.json`，并按 URL 组织。

### 需求：知识库适配
`knowledge_base.py` SHALL 能够处理更大规模的文档数据，支持增量添加文档。

### 需求：证据链验证适配
`reference_validator.py` 和 `agent6_verifier.py` SHALL 能够从更大规模的文档中检索相关证据。

## 移除需求
无移除需求。

## 技术约束
- 使用 Crawl4AI 的递归爬取功能
- 爬取深度限制为 3 层，避免过度爬取
- 单个页面爬取超时设置为 30 秒
- 总爬取时间限制为 10 分钟
- 内存使用限制为 2GB

## 验收标准
- 爬取的文档页面数量增加 5-10 倍
- "证据缺失"问题减少 50% 以上
- 爬取过程稳定，无崩溃或超时
- 其他组件能够正常处理更大规模的文档数据
