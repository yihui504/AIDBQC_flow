# 实战验证 L1 门控放宽 + 多样性增强 Spec

## 为什么

刚刚完成了两项关键改进：
1. **L1 门控放宽**：维度不匹配测试现在允许执行，可生成 Type-1 缺陷
2. **Prompt 多样性增强**：新增策略 6-8 和 Bug Type Coverage

需要通过实战运行验证这些改进的实际效果。

## 变更内容

### 变更：实战验证运行

- **配置**: max_iterations=6，使用本地文档
- **验证点**:
  1. L1 门控是否正确放宽（维度不匹配测试能执行）
  2. 测试多样性是否提升（缺陷类型分布）
  3. 缺陷产出数量是否增加

## 影响范围

- **运行配置**: `.trae/config.yaml`
- **产出**: GitHub Issues + 运行日志

## ADDED 需求

### 需求：实战验证运行

系统 SHALL 执行一次完整的实战运行以验证改进效果。

#### 场景：验证 L1 门控放宽
- **WHEN** 运行测试生成和执行流程
- **THEN** 维度不匹配测试能执行并生成缺陷
- **AND** 日志中应出现 "L1 Warning" 而非 "L1 Gating Failed"

#### 场景：验证测试多样性
- **WHEN** Agent2 生成测试用例
- **THEN** 测试应覆盖更多维度和 bug 类型
- **AND** 缺陷分布应更均匀

## 验收标准

1. 运行成功完成，无崩溃
2. 产出 GitHub Issues 数量 >= 之前运行的 8 个
3. 日志显示 L1 门控放宽生效
4. 缺陷类型分布更广泛
