# 项目下一步开发改进建议 Spec

## Why

基于对项目全部核心源码（9 个 Agent、状态管理、适配器层、配置系统、Dashboard 框架、ROADMAP）的深度审查，结合 v4.5 验证数据（Type-4 占比 91.7%，Type-1/Type-3 为 0%），识别出以下关键改进机会：

1. **核心能力短板**: 缺陷类型分布严重失衡，Type-1/Type-3 检测率为零，影响工具完整性和学术价值
2. **工程化缺口**: Dashboard 框架存在但功能空壳；无自动化回归测试；无 CI/CD 集成
3. **代码质量隐患**: Agent5 硬编码容器名、Dashboard 导入不存在的模块、AGENTS.md 版本号不一致等
4. **学术产出压力**: P0 优先级——开题报告要求学术论文，当前完成度 0%
5. **性能优化空间**: 单次运行 ~15 分钟，目标 <10 分钟

## What Changes

本文档为**纯分析建议规格**，不修改任何代码。输出物为一份结构化的改进建议报告，包含：
- 基于代码证据的优先级排序改进方向
- 每个方向的具体实施路径和技术方案
- 投入产出比（ROI）评估
- 与现有 ROADMAP 的对齐与补充

## Impact

- Affected specs: ROADMAP.md 方向细化与优先级重校准
- Affected code: 不直接修改代码，但建议涉及的关键文件已标注
- 输出: 更新后的 ROADMAP 补充建议 + 可选的实施 Task 分解

## ADDED Requirements

### Requirement: 改进建议基于代码证据

所有改进建议 SHALL 基于实际代码分析结果，包含：

#### Scenario: 建议有据可依
- **WHEN** 审查者阅读每条改进建议
- **THEN** 该建议应引用具体代码位置（文件名+行号）、验证数据或架构特征作为依据

### Requirement: 建议覆盖四个维度

改进建议 SHALL 覆盖以下维度：
1. **核心能力增强** — 缺陷检测精度、类型覆盖均衡性
2. **工程质量提升** — Dashboard 完善、回归测试、CI/CD
3. **代码健康度** — 硬编码消除、模块依赖修复、文档一致性
4. **学术与交付** — 论文产出路径、技术报告完善
