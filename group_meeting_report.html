AI-DB-QC

一个面向向量数据库（Milvus / Qdrant / Weaviate）的自动化质量检测流水线：从文档提炼契约、生成对抗测试、双层门控执行、预言机校验、缺陷去重与可复现验证，最终沉淀为可提交的标准化 Issue。

目标 让“可复现的高价值 DB 缺陷”更快浮出水面
关键词 L1/L2 门控 · 三层契约 · 多智能体 · 预言机
生成日期 2026-04-15

痛点：向量数据库测试为什么难

Problem
Speaker Notes
  • 传统单测依赖“明确预期”，但向量检索很多性质是统计/语义的。
  • 纯 fuzzing 产出多，但大量是环境/前置条件问题，难以沉淀成高价值缺陷。
  • 向量 DB 的行为约束分散在文档、API schema 与实现细节中，人工提炼成本高。
  • 对外提交 issue 需要证据链：复现、日志、断言、版本、与“官方契约”引用。
核心矛盾
“生成”容易，“验证与沉淀”难
需要把随机探索结果变成可复现、可解释、可提交的缺陷证据包。
关键要求
分层门控 + 分层预言机
先判断“请求是否合理 / 环境是否就绪”，再判断“结果是否违反数学/语义契约”。
交付目标
标准化 GitHub Issue
每条 issue 有明确断言、最小复现、日志证据与可追溯的契约来源。

方案：多智能体流水线 + 契约驱动

Solution
Speaker Notes
  • 把“自动化找 bug”建模为一条可回放、可验证、可产出 issue 的闭环流水线。
  • 用 WorkflowState 做唯一事实源，所有 agent 以“状态子集更新”协作。
  • 用三层契约（L1/L2/L3）约束生成与执行，让结果能被解释与追溯。
  • 用 L1/L2 双门控减少误报，把“环境问题”从“DB bug”中剥离。

核心思想：先把“什么算 bug”写成契约与预言机，再让生成与执行围绕契约持续探索，并把误报压到最少。

核心机制：门控、契约、预言机、验证

Mechanisms
Speaker Notes
  • L1：参数/抽象合法性；L2：运行时就绪性。两者用于“把环境误报隔离出去”。
  • 三层契约：L1 API 强约束、L2 Runtime 状态约束、L3 语义/业务约束。
  • 预言机分层：传统（数学/一致性）+ 语义（LLM/语义相似）协同。
  • 最终产出必须经过去重与隔离复现，且 fail-closed（安全默认拒绝 host 执行）。
增强图：L1/L2 门控如何影响缺陷类型与误报
三层契约（L1/L2/L3）：把“什么算 bug”写出来

L1（API 强约束）用于过滤/标记非法参数；L2（Runtime 状态约束）用于避免“未建库/未插入/未加载”等环境误报；L3（语义/业务契约）描述更高层的相关性、一致性与业务正确性。

预言机：传统 + 语义分层校验

传统预言机更适合稳定的数学性质（范围、单调性、一致性、Top-K 等）；语义预言机用于相关性/语义一致性等更难用硬规则表达的性质。二者结合，使“发现”与“解释/提交”可以兼顾。

验证与去重：把产出变成可提交的 Issue

去重用于合并“同根因不同表述”的缺陷，验证用于把候选缺陷转换为可复现结论，并且隔离执行默认 fail-closed（禁止 host fallback），避免安全与误判风险。

def should_continue_fuzzing(state: WorkflowState) -> str:
    if state.total_tokens_used >= state.max_token_budget:
        print(f"[Circuit Breaker] Token budget exceeded ({state.total_tokens_used}/{state.max_token_budget}). Terminating fuzzing.")
        state.should_terminate = True
    if state.should_terminate:
        return "verify"
    return "fuzz"

def _l1_gating(self, tc: TestCase, state: WorkflowState) -> tuple[bool, Optional[str], Optional[Dict[str, Any]]]:
    if not state.contracts:
        return False, None, None
    l1_contract = state.contracts.l1_api if hasattr(state.contracts, 'l1_api') else state.contracts.get('l1_api', {})
    dc = l1_contract.get("dimension_constraint")
    allowed_dimensions = l1_contract.get("allowed_dimensions", [])
    if dc:
        dc_obj = DimensionConstraint(**dc) if isinstance(dc, dict) else dc
        if not dc_obj.contains(dimension):
            hard_block = self._is_l1_hard_block_enabled()
            violation_details = {"violation_type": "dimension_out_of_range", "actual_value": dimension, "blocked": hard_block}
            return (not hard_block), warning, violation_details
    elif allowed_dimensions and dimension not in allowed_dimensions:
        hard_block = self._is_l1_hard_block_enabled()
        violation_details = {"violation_type": "dimension_out_of_range", "actual_value": dimension, "blocked": hard_block}
        return (not hard_block), warning, violation_details
    return True, None, None

def _l2_gating(self, tc: TestCase, state: WorkflowState) -> tuple[bool, Optional[str]]:
    collection_name = getattr(state, 'current_collection', None)
    if not collection_name:
        return False, "No active collection"
    has_data = getattr(state, 'data_inserted', False)
    if not has_data:
        return False, "Collection empty - no data inserted"
    return True, None

验证：从候选缺陷到可提交 Issue

Validation
Speaker Notes
  • 组会版只展示每库 1 个代表性案例，强调“证据链与可复现”。
  • Issue 筛选要避免 L2 门控/环境误报与 harness 假设被当成官方约束。
  • 最终交付物是 issues/ 下的 Markdown，可直接作为上游提交素材。
  • 后续工作可沿着：覆盖率→更强验证→更稳去重→更自动的官方契约对齐推进。
Issue 筛选原则（摘要）
  • 优先：可复现且自洽（MRE 能跑、断言明确、证据具体）。
  • 优先：引用官方契约，而不是仅引用 harness 生成的假设约束。
  • 优先：Actual Behavior 有具体证据（日志、返回片段、数值）。
  • 排除：L2 门控/环境错误伪装成数据库 bug（如 “No active collection”）。
